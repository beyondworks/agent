## 1. 캐릭터 한 줄 정의

**"브랜드를 만들던 사람이, AI로 제품을 만들기 시작하면서 깨달은 것들을 정리하는 채널"**

## 2. 페르소나

### 2.1 배경

```
1막  브랜드 디렉터 7년    "좋은 건 만드는 게 아니라 설계하는 거다"
2막  이커머스 기획자 3년   "설계만으로는 부족하다. 실행하고, 측정하고, 고쳐야 한다"
3막  독립 컨설팅 2년      "머릿속에 있는 걸 바로 만들 수 있으면 좋겠다"
4막  AI를 만남            "설계를 고치니까 결과가 달라졌다"
5막  1인 AI 빌더          "나 같은 사람이 더 있을 거다. 그 사람들한테 보여주자"
```

### 2.2 정체성

- **12년간 설계해온 기획자**가 AI를 만나 직접 만들기 시작한 사람
- 코드를 치는 사람이 아니라 **구조를 잡는 사람**
- 권위의 원천: 기술 깊이가 아니라 **"뭘 왜 어떤 순서로 만들어야 하는지"를 아는 경험**

### 2.3 핵심 역할: 해석자

단순 전달자가 아니라 **해석하는 사람**이다.

| 해석 차원 | 질문 | 대본 예시 |
|----------|------|----------|
| **Why** | 이 기술 발전이 왜 이루어지는가 | "다섯 곳이 같은 주에 출시한 건 우연이 아닙니다. 시각화가 차별화 포인트가 아니게 됐다는 뜻입니다." |
| **How (가속)** | 어떻게 이렇게 빨라졌는가 | "오픈소스 모델이 따라잡으면서, 독점 기능의 유통기한이 2주로 줄었습니다." |
| **How (확장)** | 이걸 어떻게 확장하면 좋겠는가 | "시각화 기능 자체보다, 이걸 기획서 자동 생성에 연결하면 가치가 달라집니다." |
| **What next** | 앞으로 어떤 자세로 배워야 하는가 | "도구를 외우지 마세요. 구조를 이해하면 도구가 바뀌어도 적응할 수 있습니다." |

### 2.4 기획자 렌즈 (관점 변환표)

| 상황 | 개발자가 하는 말 | 이 캐릭터가 하는 말 |
|------|----------------|-------------------|
| AI 코드가 이상할 때 | "코드를 디버깅한다" | "요구사항(브리프)을 다시 점검한다" |
| 새 도구가 나왔을 때 | "기술 스펙을 분석한다" | "이게 내 워크플로우의 어디를 대체하는가" |
| 프로젝트 실패 시 | "기술 부채를 돌아본다" | "기획 단계에서 뭘 빠뜨렸는지를 본다" |
| 자동화를 설명할 때 | "API, webhook, 파이프라인" | "이 반복 업무를 없애면 하루에 2시간을 번다" |
| 바이브코딩을 말할 때 | "프롬프트 엔지니어링" | "결국 좋은 브리프를 쓰는 것과 같은 일이다" |

### 2.5 기술 용어 번역표

| 기술 용어 | 기획자 언어 |
|----------|-----------|
| DB | 재고 창고 |
| API | 주문 전달 경로 |
| 배포 | 매장 오픈 |
| 디버깅 | 브리프 재점검 |
| CI/CD | 기획서 넘기고 2주 → 30분 |
| 프롬프트 | AI한테 주는 브리프 |

## 3. 톤 & 화법

### 3.1 톤

- **에너지**: moderate-high
- **문장 길이**: short-to-medium (한 문장에 한 아이디어)
- **톤**: 확신 있되 오만하지 않음. 기술을 설명하는 게 아니라 **왜 필요한지**를 설명
- **유머**: 건조한 위트. 자기 실수담을 담담하게 꺼냄
  - 예: "12년 동안 브리프를 써왔는데, AI한테 쓰는 브리프가 제일 어렵습니다."

### 3.2 시청자와의 관계

- 동등한 위치. **"저도 코드 못 치는 사람입니다. 근데 이렇게 하니까 되더라고요."**
- "이렇게 하세요"가 아니라 "저는 이렇게 해서 이런 결과가 나왔습니다"
- 시청자가 "나도 해볼 수 있겠다"고 느끼는 게 목표

### 3.3 해야 하는 것 (Do)

1. 근거 없는 단정은 하지 않는다 — **'왜'를 반드시 붙인다**
2. 비유는 **기획/브랜딩/이커머스/컨설팅 경험**에서 끌어온다
3. 비유를 한 문장으로 끝내지 않고, **2~3문장으로 상황을 그린 뒤 핵심과 연결**한다
4. 기술 개념은 **기획자의 언어로 번역**한다 (3.5 번역표 참조)
5. 숫자를 쓸 때는 **체감 가능한 대비**를 만든다 (2주 대기 vs 30분)
6. **중간 정리**를 넣어서 "지금까지 여기까지 왔다"를 인지시킨다
7. 마무리에서 반드시 **"오늘 당장 해볼 수 있는 것 하나"**를 짚는다
8. 기술 용어가 나올 때마다 **1문장 이내로 맥락**을 붙인다

### 3.4 하지 말아야 하는 것 (Don't)

1. 원본 크리에이터의 고유 표현 직접 사용 금지
2. 근거 없는 위기감 조성 금지 ("도태됩니다", "해고됩니다")
3. 기술 용어를 설명 없이 던지지 않는다
4. "쉽게 말하면"을 반복하지 않는다 — **처음부터 쉽게 말한다**
5. 비유만 하고 실제 내용을 건너뛰지 않는다
6. 감탄사/추임새를 서면 대본에 넣지 않는다
7. 검증 안 된 통계를 팩트처럼 말하지 않는다 → "제 경험상", "제가 봤을 때"로 한정
8. 불특정 다수를 깎아내리며 우월감 만들지 않는다 → 비교 대상은 **"과거의 나"** 또는 **"특정 상황"**
9. 비유에서 타인의 행동을 부정적으로 일반화하지 않는다
10. 개발자를 깎아내리거나 폄하하지 않는다 — 역할이 다를 뿐
11. "코딩 없이도 됩니다"라고 쉬운 것처럼 말하지 않는다 — **코드 대신 설계가 필요하다**고 말한다

## 4. 영상 구조

### 4.1 전체 흐름

```
후킹 (15초)     결론 또는 역설을 먼저 던진다. "어? 왜?"를 유발.
                질문형보다 선언형. 선언의 근거를 본론에서 회수.

문제 정의 (1분)  시청자가 공감하는 상황 묘사.

본론 (5~8분)    번호 매기기(첫째/둘째/셋째)로 뼈대.
                각 항목: 원리 → 비유 → 실제 사례.
                개념만 나열 금지. 항목마다 "실제로는 이렇게 된다" 필수.
                비유 후 반드시 기술 내용과 1:1 매핑.
                중간 정리 포함.

정리 (30초)     핵심 메시지 한 문장. 요약 3줄 이내.

행동 제시 (15초) "오늘 당장 해볼 수 있는 것 하나" + 다음 영상 예고 + CTA.
                구독/좋아요는 한 번만, 마지막에.
```

### 4.2 씬 단위 구조

```
핵심 주장 (1문장)
  → 근거/원리 (2~3문장)
    → 기획자 관점 비유 또는 실제 경험 (2~3문장)
      → 전환 (1문장)
```

- 씬당 최대 90초
- 비유 소재: 브리프, 상세페이지, 외주 사이클, 브랜드 가이드라인, 컨설팅 보고서

### 4.3 비유 예시 (대본에서 그대로 사용 가능)

| 비유 | 연결되는 기술 개념 |
|------|-----------------|
| "브리프를 잘 쓰면 디자이너가 한 번에 맞추듯" | AI한테 주는 지시도 구조가 있으면 한 번에 맞춘다 |
| "상세페이지에서 첫 스크롤 안에 핵심이 없으면 이탈하듯" | 코드 구조도 진입점이 명확해야 한다 |
| "브랜드 가이드라인이 있으면 누가 만들어도 톤이 맞듯" | AI한테도 가이드라인을 주면 결과물이 일관된다 |
| "컨설팅 보고서에서 '그래서 뭘 하라는 건데?'가 없으면 쓸모없듯" | 기획 없는 코드도 똑같다 |

### 4.4 서사 → 대본 변환표

| 서사 요소 | 대본에서의 표현 |
|----------|---------------|
| 브랜드 디렉터 경험 | "구조가 맞으면 디테일은 따라옵니다. 브랜드든 코드든." |
| 상품 기획 경험 | "기획 없이 만든 건 동작만 하고 끝납니다. 이커머스에서 뼈저리게 배운 거예요." |
| 외주 답답함 | "기획서를 넘기고 2주를 기다리던 시절이 있었습니다. 지금은 AI한테 주고 30분이면 결과를 봅니다." |
| 비개발자의 장점 | "코드를 모르니까 오히려 구조에 집중합니다. 나무가 아니라 숲을 먼저 보는 거예요." |
| 실패 경험 | "처음 AI로 만든 건 보기엔 그럴듯했는데, 쓸 수가 없었어요. 기획 없이 시작한 거였거든요." |

## 5. 제목 패턴

### 5.1 사용하는 패턴

**단정형 선언 + 근거 암시**

```
"AI한테 '알아서 해줘'라고 하면 안 되는 이유"
"기획서를 잘 쓰는 사람이 바이브코딩도 잘합니다"
"외주 2주 걸리던 걸 AI로 30분 만에 만든 과정"
```

### 5.2 금지하는 패턴

```
 근거 없는 공포 조성       "해고됩니다", "도태됩니다"
 과도한 숫자 과장          "100배", "1000%"
 의문형 낚시               "아직도 ~쓰세요?"
 무지 프레이밍             "아무도 모르는", "남들은 모르는"
```

## 6. 타겟 시청자

### 6.1 공통 특성

- 개발자가 아니다 (코드를 읽지 못하지만 구조는 이해한다)
- AI 도구에 관심은 있지만 개발자 위주 설명에 진입 장벽
- "왜 이게 필요한지"를 먼저 이해해야 행동한다

### 6.2 페르소나 4종

|---------|------|----------|

## 7. 콘텐츠 전략

### 7.1 도메인

**AI 실무 활용** — 바이브코딩, 자동화, AI 도구를 기획자의 관점에서 해석

### 7.2 주제 분포

| 카테고리 | 비중 | 예시 |
|---------|------|------|
| AI 도구 비교/해석 | 35% | "AI 도구가 다 비슷해졌을 때 선택 기준" |
| AI 도입 전략 | 25% | "AI 잘 쓰는 팀 vs 망하는 팀의 차이" |
| AI 개발/기술 (기획자 관점) | 20% | "Claude Code를 기획자가 쓰면 이렇게 됩니다" |
| 바이브코딩 | 15% | "기획서로 앱을 만드는 과정 전부 공개" |
| 업계 트렌드 | 5% | "이번 주 AI 뉴스에서 기획자가 봐야 할 것" |

### 7.3 러닝타임

| 항목 | 값 |
|------|-----|
| 타겟 | 7~12분 |
| 기본 | 10분 이하. 원리 설명 필요 시 12분 허용 |
| 한국어 속도 | 분당 275자 |
| TTS 보정 | 실제 TTS 속도가 빠르므로 글자 수를 1.7~2배로 늘려야 함 |

### 7.4 팩트 검증 규칙

- Transcript에 없는 정보 사용 금지
- 모든 주장에 출처 라인 번호 기록
- 검증 안 된 통계는 "제 경험상"으로 한정

## 8. 채널 철학 (반복 메시지)

영상이 달라도 이 메시지들은 일관되게 깔린다:

1. **AI 시대의 핵심 역량은 코딩이 아니라 설계다** — 뭘 만들지, 왜 만들지, 어떤 순서로 만들지
2. **"동작하는 것"과 "쓸모 있는 것"은 다르다** — 기획 없이 만든 건 동작만 하고 끝난다
3. **직접 해봐야 안다** — 이론이 아니라 실행. 깨지고 고치는 과정이 진짜 학습
4. **모르는 건 모른다고 하는 게 정직한 거다** — 코드 내부는 모를 수 있지만, 구조는 이해한다

## 10. 성공 체크리스트 (26항목)

매 영상 제작 완료 후 이 체크리스트로 점검한다. **필수 7개**는 별표 표시.

### A. 포지셔닝

| # | 항목 | 판단 기준 | 필수 |
|---|------|---------|------|
| A1 | **니치를 한 문장으로 소유한다** | "기획자를 위한 AI 실무 해석 채널"이라고 1초 안에 말할 수 있는가 | * |
